## Версионирование

- Префикс пути: `/api/v1/…`
- Breaking changes — только в `/api/v2/`; v1 поддерживается до миграции клиентов (bot, web).
- Non-breaking: новые optional-поля, новые endpoint'ы в v1.

## Аутентификация (MVP)

| Механизм | Описание |
|----------|----------|
| `Authorization: Bearer <token>` | Service token bot → backend; env: `BACKEND_SERVICE_TOKEN` (task-06 в `.env.example`) |
| `telegram_id` в теле/query | Идентификация пользователя-диабетика; маппинг на Telegram `chat.id` |

Web-auth и роли доктора — позже.

## Заголовки

| Заголовок | Обязательный | Описание |
|-----------|--------------|----------|
| `Authorization` | да (кроме `/health`) | Bearer service token |
| `X-Request-Id` | нет | UUID трассировки; backend echo в логах |
| `Content-Type` | да для POST | `application/json` |

## Именование JSON

- Поля: `snake_case`
- UUID: строки RFC 4122
- Время: ISO 8601 UTC (`…Z`)
- Доменные сокращения в API: `xe` (ХЕ), `bje` (БЖЕ) — см. [data-model.md](../data-model.md#api-поля-v1)

## Лимиты тела запроса (MVP)

| Поле / ресурс | Рекомендация | Превышение |
|---------------|--------------|------------|
| `image_base64` (сценарий A) | ≤ 5 MB после декодирования | **413** `PAYLOAD_TOO_LARGE` (post-MVP) |
| JSON body (прочее) | разумный размер для bot-клиента | **413** (post-MVP) |

На MVP backend может отклонять слишком большие payload до введения формального 413 — логировать на сервере.

## Формат ошибки

Единое тело для всех кодов, кроме случаев, когда FastAPI отдаёт стандартный `422` (см. ниже):

```json
{
  "error": {
    "code": "NOT_FOUND",
    "message": "Краткое описание для клиента",
    "details": {}
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `code` | string | машинный код (`UPPER_SNAKE_CASE`), стабильный для клиентов |
| `message` | string | человекочитаемое сообщение; без внутренних stack trace |
| `details` | object \| array | опционально: поля валидации, id ресурса, retry-after |

**422 (валидация):** допускается стандартный формат FastAPI:

```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "telegram_id"],
      "msg": "Field required",
      "input": {}
    }
  ]
}
```

Клиенты (bot) должны обрабатывать оба формата на MVP; целевое — маппить `422` в понятное сообщение пользователю.

## HTTP-коды ошибок

### Используемые на MVP

| HTTP | Код `error.code` | Когда |
|------|------------------|-------|
| **400** | `BAD_REQUEST` | Семантически неверный запрос (не синтаксис JSON): пустой текст и фото одновременно, недопустимая комбинация полей |
| **401** | `UNAUTHORIZED` | Нет или неверный идентификатор клиента: отсутствует `telegram_id`, неверный service token bot→backend |
| **403** | `FORBIDDEN` | Доступ к чужому ресурсу: диалог/событие другого пользователя |
| **404** | `NOT_FOUND` | Нет ресурса: пользователь, диалог, запрос, событие по id |
| **422** | *(FastAPI `detail`)* | Ошибка схемы: тип, обязательные поля, формат UUID, диапазоны |
| **502** | `LLM_UNAVAILABLE` | OpenRouter недоступен, таймаут или ошибка провайдера после retry |
| **503** | `SERVICE_UNAVAILABLE` | PostgreSQL недоступна или backend не готов (health degraded) |
| **500** | `INTERNAL_ERROR` | Необработанное исключение; без деталей наружу |

### Зарезервированы (после MVP)

| HTTP | Код | Когда |
|------|-----|-------|
| **409** | `CONFLICT` | Конфликт состояния: дубликат, параллельное изменение |
| **429** | `RATE_LIMITED` | Превышен лимит запросов; в `details` — `retry_after_sec` |
| **413** | `PAYLOAD_TOO_LARGE` | Фото/тело превышает лимит |

## Маппинг по сценариям (MVP)

### Сценарий A — вопрос ассистенту

| Ситуация | HTTP |
|----------|------|
| Нет `telegram_id` | 401 |
| Нет текста и медиа | 400 |
| Невалидное тело (тип полей) | 422 |
| Диалог не найден | 404 |
| LLM/OpenRouter ошибка | 502 |
| БД недоступна | 503 |

### Сценарий B — фиксация события

| Ситуация | HTTP |
|----------|------|
| Нет `telegram_id` | 401 |
| `request_id` / `food_event_id` другого пользователя | 403 |
| `request_id` / `food_event_id` не найден | 404 |
| Невалидные ХЕ/тип события, `dose` ≤ 0 | 422 |
| Невалидный `from`/`to` (GET food) | 422 |
| БД недоступна | 503 |

## Правила для реализации (backend)

- Не возвращать **401** для «ресурс не найден» — только **404**.
- **403** — только когда ресурс существует, но пользователь не владелец (не путать с 404 для скрытия существования — на MVP допустим 404 для чужих id).
- **502** — только для внешних интеграций (OpenRouter); **503** — для инфраструктуры backend/БД.
- Логировать причину **500/502/503** на сервере; в `message` — нейтральный текст для пользователя.
- Не логировать и не отдавать в `details` промпты, тексты сообщений, токены.

## Правила изменений

1. Новый код ошибки — добавить в этот документ и в тесты (task-04).
2. Изменение `error.code` или семантики HTTP — breaking change → bump версии API.
3. OpenAPI (`/docs` или `docs/api/openapi.yaml`) должен совпадать с таблицами выше.
