# Техническое видение проекта

Опирается на [идею проекта](idea.md).

## Технологии

| Компонент | Выбор |
|-----------|--------|
| Язык | Python 3.12+ |
| Окружение | `venv` (стандартная библиотека) |
| Зависимости | `uv` — установка и lock-файл (`uv sync`) |
| Telegram | `aiogram` 3.x, long polling |
| LLM | OpenAI-compatible клиент (`openai`), провайдер **OpenRouter** |
| Линт / формат | `ruff` |
| Автоматизация | `Makefile`: `install`, `run`, `lint`, `format` |

Отдельная сборка не требуется — интерпретируемый Python, запуск из venv.

## Принципы разработки

- **KISS** — минимум слоёв и абстракций, без преждевременной оптимизации.
- **Один класс — один файл** — каждый значимый класс в отдельном модуле.
- **Явная конфигурация** — секреты и настройки через переменные окружения, не в коде.
- **Тонкий бот, умная LLM** — логика диалога и роль задаются системным промптом; бот маршрутизирует сообщения и вызывает API.
- **MVP без БД** — состояние диалога в памяти процесса; персистентность откладывается.

## Архитектура

```
Пользователь (Telegram)
        │
        ▼
   aiogram polling
        │
        ▼
   Handlers ──► SessionStore (история по chat_id)
        │
        ▼
   LlmClient ──► OpenRouter API
        │
        ▼
   Ответ в Telegram
```

Поток обработки: сообщение или фото → handler → история из `SessionStore` → запрос к LLM (system + history + user) → ответ пользователю → обновление истории.

Фото: handler скачивает файл через Telegram API и передаёт в LLM как image (vision-модель через OpenRouter).

## Модель данных / состояние

**В памяти процесса** (`SessionStore`):

- `chat_id` → список сообщений `{role, content}` (последние N пар, чтобы не раздувать контекст).
- Для фото: multimodal content (base64 или URL) в составе user-сообщения.

**Не храним в MVP:** профили пользователей, долгосрочный дневник ХЕ/инсулина, БД.

Учёт ХЕ и инсулина — через диалог и ответы LLM, без отдельной схемы данных.

## Работа с LLM

- **Провайдер:** OpenRouter (`https://openrouter.ai/api/v1`).
- **Клиент:** `openai.OpenAI(base_url=..., api_key=...)`.
- **Модель:** задаётся через `LLM_MODEL` в env; должна поддерживать vision, если пользователь присылает фото.
- **Системный промпт:** файл `prompts/system.txt`, загружается при старте через `Prompt`. Роль помощника диабетика: оценка ХЕ, питание, инсулин; дисклеймер «не замена врачу»; тон поддерживающий, без назначения доз.
- **Запрос:** `messages` = system + history + user; для фото — multimodal content.
- **Ограничения:** лимит длины истории (`LLM_MAX_HISTORY`); таймаут и обработка ошибок API с понятным сообщением пользователю.

Вопросы по разработке и Generative AI — из параметрической памяти модели, без RAG и внешних файлов курса.

## Сценарии работы пользователя

1. **Текстовый вопрос** — пользователь пишет → бот отвечает с учётом истории чата.
2. **Фото еды** — пользователь отправляет фото → бот оценивает ХЕ / состав с оговорками.
3. **Описание еды текстом** — «борщ и хлеб» → оценка ХЕ.
4. **Инсулин** — «вколол 4 единицы» → обсуждение в контексте еды (без назначения доз).
5. **Смешанный диалог** — несколько реплик подряд, контекст сохраняется в сессии.
6. **Команда /start** — приветствие и краткая инструкция.

## Конфигурация и секреты

Переменные окружения (`.env`, образец в `.env.example`):

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | токен бота |
| `OPENROUTER_API_KEY` | ключ OpenRouter |
| `LLM_MODEL` | идентификатор модели на OpenRouter (обязательно) |
| `LLM_MAX_HISTORY` | число пар сообщений в контексте (по умолчанию 10) |
| `LOG_LEVEL` | уровень логирования |

`.env` в `.gitignore`. Секреты не коммитить.

## Логирование

- Стандартный `logging`, настройка при старте в `main.py`.
- Уровень из `LOG_LEVEL`.
- Логировать: старт/остановка, `chat_id` входящих сообщений, ошибки API, таймауты.
- На уровне INFO не логировать полный текст сообщений и промптов.
- Токены и ключи не логировать.

## Сборка и запуск

**Makefile:**

- `make install` — `uv venv` + `uv sync`
- `make run` — запуск `python -m diaai.main` из venv
- `make lint` — `ruff check`
- `make format` — `ruff format`

**Локально:** скопировать `.env.example` → `.env`, заполнить переменные, выполнить `make install && make run`.

**Деплой:** опционально, отложен. Варианты (VPS + systemd, Docker, PaaS) не фиксируются в MVP.
